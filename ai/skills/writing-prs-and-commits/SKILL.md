# Writing PRs and commits

Use this skill when the user asks to create a PR, update an existing PR title or body, or write a commit message for a change that maps cleanly to one PR.

## Resolve the target

- If the user names a PR number or URL, use that.
- Otherwise, inspect the current branch and use `gh pr view --json number,title,body,baseRefName,headRefName,url` to find the associated PR.
- If no PR exists and the user wants one created, create it as a **draft with no reviewers** unless they ask otherwise (see PR creation below), then verify the title and body.
- If the user wants text only, provide the draft without mutating GitHub.
- When a PR is part of a stack, describe only the net change between the PR head and its base branch. Do not describe the full stack relative to `main`. Inspect that net change with `git diff <base>...<head>` or `gh pr diff`.

## Core writing rules

- Lead with _why_. The first paragraph or sentence should explain the problem, motivation, or user impact.
- Follow with _what_ changed. Once the reader understands why the change exists, describe the solution and any important constraints or tradeoffs.
- Write the net change only. Do not document abandoned experiments, reverted approaches, or temporary dead ends.
- Prefer natural prose over rigid `Summary` / `Problem` / `Solution` headings unless the user explicitly asks for that format.
- Keep references repo-relative. Do not mention absolute local filesystem paths.
- Preserve valuable existing PR content such as screenshots, images, rollout notes, and manually written context unless the user asks to remove it.
- Use Markdown cleanly: inline code in backticks, fenced blocks when needed, and GitHub permalinks when citing specific code.

## PR titles

Titles must follow the `$type/$domain: $title` format.

- `$type` is a conventional-commit type: `feat`, `fix`, `chore`, `refactor`, `docs`, `test`, or `perf`.
- `$domain` is the area of the codebase affected, e.g. `auth`, `mcp`, `api`, `dev`.
- `$title` is imperative mood, lowercase, with no trailing period.

Examples:

- `fix/auth: reject expired refresh tokens`
- `feat/mcp: add resource subscription support`
- `refactor/api: collapse duplicate error handlers`

## PR descriptions

- Write the opening prose in this order:
  1. Why this change is needed.
  2. What changed.
  3. Any important tradeoffs, rollout notes, or follow-up context.
- When relevant, reference linked Linear issues or related Slack threads.
- If docs or handbook follow-up is needed, mention it in the prose before the verification section.
- Include `## Test Plan` near the end only when there is meaningful validation to report.
- The test plan should mention intentional verification such as focused tests, manual flows, or reproduction steps. Do not pad it with routine formatting or CI work the repository already expects.
- Do not add a Test Plan merely because the existing PR lacks one, because a template might ask for one, or to be "honest" about not running local checks. Placeholder entries like "Not run locally; CI will validate this" are not meaningful validation and should be omitted.

## Creating and editing PRs

Apply the exact crafted text — never rely on the interactive editor, which discards the draft.

- Create: `gh pr create --draft --title "..." --body-file <file>` (or `--body "..."`). Default to `--draft` and no reviewers unless the user asks otherwise.
- Edit: `gh pr edit <number> --title "..." --body-file <file>`. To pipe a body from stdin, use `--body-file -`.
- For multi-line bodies, prefer `--body-file` with a heredoc or temp file over `--body` so formatting and Markdown survive intact.

## Commit messages for single-PR changes

Treat a single commit that is intended to become one PR as a PR — GitHub uses the commit title and body by default. Apply the same rules: explain _why_, not _what_ (the diff already shows the what), and use the conventional-commit prefix from the title format above (e.g. `fix/auth:`).

## Workflow

1. Inspect the diff, linked context, and any existing PR title, body, or commit message.
2. Decide whether you are creating a new PR, editing an existing PR, or writing a stand-alone commit message.
3. Draft the title and body so the motivation is clear before the implementation details.
4. Check that the text matches the net change and preserves valuable existing content. Omit any section (such as the Test Plan) when it would only be boilerplate.
5. Apply the final text with `gh pr create`, `gh pr edit`, or `git commit` as requested.

## Attribution

Never add AI attribution to a commit or PR — no "Generated by Claude Code" lines, no `Co-Authored-By` trailers, and no similar tool credits.
